6 个改动 · 风险与收益逐项核对

每个瓶颈对应一个具体改动 · 不是无脑砍，是带保留路径的替换

1
瓶颈send_message 后强制再跑一轮 LLM
改动启发式早 break
条件叠加：fast user path + 非 voice 渠道 + 无 active task + content ≥ 8 字 + 不含 "查一下/看看/稍等/等等" 等 progress 词 + 不以问号结尾。
任一不满足 → 保留 nudge 走原路径（progress 链 / TTS 后续 / 任务流程 全部不破）。
省 1-2s · 简单回复延迟降一半
2
瓶颈主轮单次 prompt 15-17K tokens
改动system prompt 按需切片
CORE 段 5.5K 常驻（Top-Level / Posture / Response / Meaning-First / Style / 多渠道身份）。
Music / Video / WeatherCard / WeChat outbound / Focus Banner / ACUI 用关键词 P1 gate（沿用 AGENT_KEYWORD_RE 模式）。
主轮 tokens ↓ 25-35% · cache miss 部分变小
3
瓶颈focus-classifier sync 800ms 硬超时
改动全路径 async 化
TICK / background 不再阻塞当前轮 buildContextBlock。
当前轮 <focus> 看 v0 ngram topic，下一轮看 LLM patch 过的 refined topic。
代价：TICK 决策的 focus 信号粗 1 轮，影响小。
省 800ms · 每个 TICK 路径
4
瓶颈gatherContext 1-3 轮独立 LLM
改动改启发式 + 缓存
task 关键词 vs taskKnowledge 关键词覆盖度 > 0.6 → sufficient。
消息提到 .js/.py/.md 路径 → read_file need；提到 "上次/那个" → search_memory need。
漏检由主轮 LLM 自己用工具兜底（mustReply 安全网）。
省 3-6s · 任务态触发时
5
瓶颈refresh-loop 最多 6 轮 LLM
改动启发式打 confidence 分
baseMemories 数 / 期望召回 + query 关键词覆盖率 + 时间分布（陈旧扣分）+ 类型多样性。
score ≥ 0.7 直通；0.4-0.7 跑启发式 expand keywords（无 LLM）；< 0.4 才进 1 轮 LLM eval。
省 5-10s · TICK + 任务触发时
6
瓶颈embedding HTTP 800ms + 全表 cosine
改动进程内 LRU cache + 上限保护
focusText hash → 5min TTL · LRU 100 项 · 跨轮复用。
memories > 5000 且无 sqlite-vec 时跳过向量召回，FTS5-only 兜底。
scaling 防御 + 命中即省。
命中 cache 省 800ms · scaling 风险消除
